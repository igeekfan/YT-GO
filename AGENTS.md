# AGENTS.md - Guide for AI Coding Agents

## Project Overview

Wails v2 desktop application with a Go backend and React/TypeScript frontend.
The Go code handles app logic; the frontend is bundled via Vite and embedded into the binary.

## Directory Structure

```
.
├── main.go              # Entry point, Wails app config
├── app.go               # App struct, methods bound to frontend
├── go.mod / go.sum      # Go module (Go 1.23.0, Wails v2.12.0)
├── wails.json           # Wails project config
├── frontend/
│   ├── src/             # React + TypeScript source
│   ├── wailsjs/         # Auto-generated Wails bindings (do not edit)
│   ├── package.json     # npm scripts and dependencies
│   ├── vite.config.ts   # Vite build config
│   └── tsconfig.json    # TypeScript strict config
└── build/               # Platform-specific build assets
```

## Build / Dev / Test Commands

### Full Application
| Command | Description |
|---------|-------------|
| `wails dev` | Run dev server with hot reload (Go + frontend) |
| `wails build` | Build production binary to `build/bin/` |
| `wails build -debug` | Build with devtools enabled |

### Go Backend
| Command | Description |
|---------|-------------|
| `go build ./...` | Compile all Go packages |
| `go vet ./...` | Run Go static analysis |
| `go test ./...` | Run all Go tests |
| `go test -run TestName ./path` | Run a single Go test by name |
| `gofmt -w .` | Format all Go files |
| `goimports -w .` | Format and fix imports |

### Frontend
| Command | Description |
|---------|-------------|
| `npm run dev` | Start Vite dev server only |
| `npm run build` | Type-check (`tsc`) then build with Vite |
| `npm run preview` | Preview production build |

All frontend commands run inside the `frontend/` directory.

## Code Style - Go

- **Formatting**: Use `gofmt` (tabs for indentation, no exceptions).
- **Imports**: Standard library first, blank line, then third-party. Use `goimports` to auto-organize.
- **Naming**: CamelCase for exported, camelCase for unexported. Receiver names are short (1-2 chars, e.g. `a *App`).
- **Error handling**: Return `error` as the last return value. Check errors immediately; do not ignore them.
- **Comments**: Doc comments on exported types/functions. Keep comments minimal and meaningful.
- **Structs**: One struct per file preferred. Constructor pattern: `NewXxx() *Xxx`.
- **Wails bindings**: Methods on `App` struct that should be callable from the frontend must be exported (capitalized).
- **Context**: Store `context.Context` from `startup` for use in methods requiring it.

### Go Import Example
```go
import (
    "context"
    "fmt"

    "github.com/wailsapp/wails/v2"
    "github.com/wailsapp/wails/v2/pkg/options"
)
```

## Code Style - TypeScript / React

- **Formatting**: No semicolons at end of statements (matches existing code style).
- **Components**: Functional components with hooks. No class components.
- **Imports**: Group order - React, external libs, internal modules, styles. No semicolons after import lines.
- **State**: Use `useState` hook. Destructure tuple: `const [val, setVal] = useState(initial)`.
- **Event handlers**: Inline arrow functions or named function declarations. Parameter types: use `any` for simple DOM events, explicit types elsewhere.
- **Naming**: PascalCase for components, camelCase for functions/variables. CSS class names use kebab-case.
- **CSS**: Separate `.css` files per component. Import at top of file. Use plain CSS (no CSS modules or styled-components).
- **Exports**: Use `export default` for the main component in a file.
- **Wails calls**: Import from `../wailsjs/go/main/App`. These are auto-generated; do not modify `wailsjs/` directory.

### TypeScript Import Example
```typescript
import {useState} from 'react'
import {Greet} from "../wailsjs/go/main/App"
import './App.css'
```

## Key Conventions

- **Do not edit `frontend/wailsjs/`** - This directory is auto-generated by `wails dev` or `wails build`.
- **Frontend dist is embedded**: The `//go:embed all:frontend/dist` directive in `main.go` bundles the built frontend. Always run `npm run build` before `wails build`.
- **Binding Go methods**: Add the method to `App` in `app.go`, then add it to the `Bind` slice in `main.go`.
- **No test framework installed** for frontend. If adding tests, install `vitest` (pairs with Vite) and add a `test` script to `package.json`.
- **No linter configured**: Run `go vet ./...` for Go. For frontend, consider adding ESLint if expanding the codebase.
- **TypeScript strict mode is enabled** (`"strict": true` in tsconfig). Avoid `any` where possible; use proper types.

## Code Quality Requirements

**IMPORTANT**: After every code change, you MUST verify that the code compiles without errors:

### Frontend Check
```bash
cd frontend && npm run build
```

### Go Backend Check
```bash
go build ./...
go vet ./...
```

Always run these checks before considering a task complete. If there are compilation errors, fix them immediately before proceeding.

## Adding a New Feature

1. Add Go method to `App` in `app.go` (exported, returns value + error).
2. Add to `Bind` slice in `main.go` if not already bound.
3. Run `wails dev` to regenerate frontend bindings in `wailsjs/`.
4. Import the generated function in your React component from `wailsjs/go/main/App`.
5. Call it from your component, handle the Promise.

## Commit Rules

**After completing each independent feature or fix, commit immediately.** Do not batch multiple features into one commit.

Commit workflow:
1. `git status` — review changed files
2. `git diff` — review the diff
3. `git log --oneline -3` — check commit style
4. `git add <files> && git commit -m "<type>: <description>"`

Commit message format:
- `feat: <description>` — new feature
- `fix: <description>` — bug fix
- `docs: <description>` — documentation change
- `refactor: <description>` — refactor
- `chore: <description>` — miscellaneous

Rules:
- Each commit contains exactly one logical change
- Descriptions are concise and clear
- No period at the end of the commit message
- Always run `go build ./...` and `npm run build` before committing to confirm compilation passes

## Auto Commit & Push

**After every code change is verified, automatically run git commit and git push.** Do not wait for the user to ask.

Auto-commit workflow:
1. Code change complete + compilation verified
2. `git add <changed-files>`
3. `git commit -m "<type>: <description>"`
4. `git push`

Trigger conditions:
- After fixing a bug
- After adding a new feature
- After refactoring code
- After modifying config or documentation
- After any operation that results in file changes

Notes:
- If there are other uncommitted changes, decide whether they should be committed separately first
- Check that you are on the correct branch before pushing
- If there is a push conflict, pull first then push
- If the user explicitly says "do not push", skip the push step
