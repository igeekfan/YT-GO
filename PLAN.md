当前已实现功能统一维护在 [README.md](README.md) 与 [README.zh-CN.md](README.zh-CN.md) 中，PLAN 主要保留结构演进、未来工作项、优先级和推进顺序，避免与 README 重复维护。

最近已完成：

- 启动配置页第一步增加语言选择与明暗主题切换，首次启动即可完成界面偏好设置。
- Go 后端按职责拆分为 settings / yt-dlp / downloads / ui / diagnostics / update 多文件结构，降低单文件复杂度。

## 产品规划原则

- 设置项不再继续堆叠到单一弹窗中，后续改为按任务场景拆分入口。
- 首页优先保留“分析链接 -> 选择格式 -> 开始下载”的主流程，减少非核心配置干扰。
- 高级能力按模块逐步接入，先补最常用、最能直接提升下载成功率与可控性的功能。
- README 记录已完成能力，PLAN 只记录未来工作与演进顺序。

## 设置与界面重构方向

当前 [frontend/src/components/SettingsDialog.tsx](frontend/src/components/SettingsDialog.tsx) 承载了过多配置项，下一阶段不再继续横向堆积，而是拆为以下结构：

1. 下载设置
	默认下载目录、默认画质、文件名模板、输出容器、音频格式、限速、并发数。
2. 媒体选项
	保存描述、保存缩略图、字幕下载、字幕嵌入、章节写入、片段裁剪、SponsorBlock。
3. 网络与认证
	代理、Cookies、通用模式、额外请求相关参数。
4. 工具与维护
	yt-dlp 更新、FFmpeg 检测、诊断信息、运行时状态。
5. 外观与语言
	主题、语言、通知与界面偏好。

建议的 UI 形态：

- 第一阶段：保留设置弹窗，但改为页签或侧边栏分组。
- 第二阶段：把“下载设置”和“媒体选项”下沉到首页可见区域或折叠面板。
- 第三阶段：把“工具与维护”独立成单独页面或 About/Tools 面板。

## 分阶段实施建议

### 第一阶段：先拆设置，再补高频能力

- 完成设置分组，不再把所有选项塞进同一屏。
- 把“下载设置”和“媒体选项”做成更接近主流程的入口。
- 优先补齐字幕、章节、裁剪这些用户最容易感知的下载能力。

### 第二阶段：补齐输出控制与通用模式

- 增加输出文件名模板、输出容器、音频格式转换。
- 引入通用模式，让应用接受非 YouTube 但 yt-dlp 支持的站点 URL。
- 为特殊站点准备更明确的 Cookie、代理、额外参数入口。

### 第三阶段：完善工具与可维护性

- 将工具检测、更新、诊断从设置中剥离成独立区域。
- 增强历史记录、日志回看、失败分析与恢复能力。
- 根据后续语言扩展需要，再设计多语言资源管理方式。

## 当前推荐推进顺序

1. 先做“设置架构重构”，解决单弹窗过载问题。
2. 再做“字幕能力”，补最明显的下载功能缺口。
3. 然后做“媒体增强项”和“输出与文件控制”。
4. 最后推进“通用站点模式”和“工具中心”。
